Visão geral

O kit foi extraído do protótipo e reorganizado em fundações reutilizáveis: tokens, layout, navegação, cards, formulários, feedback visual e um runtime mínimo acionado por data-gs-*.

VisualAzul-marinho, laranja, superfícies claras, hierarquia firme e acabamento mais premium.
ComportamentoTabs, menus, toasts, modal, accordion e interações pequenas sem plugin.
EntregaHTML declarativo e consumo direto no navegador com baixa fricção para o time.
Quando usar

Use o kit como base de qualquer tela nova que precise herdar a linguagem do protótipo.

Quando não usar

Não coloque regra de negócio ou estados de produto complexos no runtime base.

Contrato central

Classes controlam forma visual. data-gs-* controla comportamento genérico.

Instalação

Inclua os arquivos abaixo e comece a compor a tela com classes do kit.

Bootstrap mínimo 
<link rel="stylesheet" href="./assets/css/germinastack.css" />
<script src="./assets/js/germinastack.js" defer></script>

<main class="gs-page">
  <section class="gs-card">Seu conteúdo</section>
</main>

Matérias em alta

JavaScript

O runtime expõe uma API pequena e previsível em window.GerminaStackUI.

  • GerminaStackUI.init(root): inicializa tabs, menus, toasts, dismiss, accordion e demos em um escopo.
  • GerminaStackUI.showToast({ title, message, tone }): cria toast programaticamente.
  • GerminaStackUI.closeMenus(): fecha todos os menus contextuais abertos.
  • data-gs-modal-open e data-gs-modal-close: ciclo de modal.
  • data-gs-accordion-trigger: controle declarativo de accordion.
Toast manual 
window.GerminaStackUI.showToast({
  title: "Deploy concluído",
  message: "A página de perfil foi atualizada com o kit.",
  tone: "success"
});

Receitas

Blocos objetivos para acelerar implementação sem obrigar o time a redescobrir a estrutura correta.

Tela nova
  • Comece com gs-page.
  • Escolha gs-stack ou grids.
  • Encapsule seções com gs-card.
Dashboard
  • Abra com gs-metric-grid.
  • Adicione progresso e tabela para leitura rápida.
  • Use timeline quando houver sequência de entrega.
Interação nova
  • Tente reaproveitar data-gs-*.
  • Se não couber, deixe o JS novo fora do runtime base.
  • Não misture regra de negócio no kit.

Acessibilidade

O kit já cobre foco visível, redução de movimento e uma base semântica razoável, mas o consumo ainda precisa seguir disciplina.

  • Mantenha button para ação e a para navegação.
  • Em tabs, use role="tablist" e role="tab" como nos exemplos.
  • Campos devem continuar com label explícita mesmo quando a UI parecer autoexplicativa.
  • Só esconda informação visualmente quando houver alternativa real para leitor de tela.
  • Para conteúdo realmente dinâmico, complemente com anúncio de estado fora do kit quando necessário.

Boas práticas

Estas regras evitam degradação visual e mantêm a coerência com o protótipo.

  • Sempre prefira sobrescrever tokens antes de criar novas cores soltas.
  • Use gs-card, gs-post e gs-side-card como blocos de composição.
  • Mantenha botões primários reservados para a ação principal da área.
  • Para novas interações, reaproveite o padrão data-gs-* em vez de amarrar seletores frágeis.
  • Em mobile, reduza quantidade de colunas antes de mexer em espaçamento e tipografia.
  • Antes de criar componente novo, valide se ele não é apenas variante de um bloco existente.
